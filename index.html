processes

Changes are proposed, discussed and implemented in the nightly compiler. They are tested there till there is agreement that the feature works and brings a benefit.

Only then the feature gets declared stable and will then be in the next stable release (which are 6 weeks apart!).

Release testing involves a crater run: A big chunk of the entire language eco-system is built and any new build failure gets investigated.

Slint

Slint defines UIs using a declarative, strongly typed, non-touring complete description language.

Slint can be used from Rust, C++ or Javascript code.

Slint code is compiled to Rust or C++ code or interpreted.

Slint scales from micro controllers to workstations.

Slint might look a bit familiar to some of you...

  • declarative: You define components out of a tree of other components that all have properties
  • strongly typed: Everything in Slin has a fixed type
  • not touring complete: No arbitrary expressions, toolable!

Slint integrates into several languages. We try to have an API that feels native in all of them.

Compiled: Low resource usage! Demo runs on <300KiB of RAM.

... and thus scales from a line buffer running on a micro controller to workstation class class machines with GPUs
